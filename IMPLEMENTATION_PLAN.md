# Поэтапный план реализации: техпроцессы ЧПУ как Telegram Mini App

Каждый этап можно сделать отдельно, закоммитить и проверить, не трогая остальное.

---

## 1. Что строим (кратко)

**Telegram Mini App** — это обычное веб-приложение, которое открывается внутри Telegram. Пользователь заходит через бота (кнопка «Открыть»), видит интерфейс: детали → техпроцесс → справочник инструмента.

**Хранилище (прод):** Google Sheets (таблицы) + Google Apps Script (API Web App) + Google Drive (фото).  
Отдельный FastAPI/PostgreSQL/VPS для MVP **не нужны**. Модули и контракты те же; меняется только реализация хранилища.

**Не входит в объём:**

- учёт смен, статусов, склада;
- режимы резания (обороты, подача, СОЖ — они в программе станка);
- патроны, центры, приспособления — только **кулачки** на токарном.

## 2. Принципы модульности и изоляции

### 2.1. Слои (снизу вверх)

```
[Google Sheets + Drive]
      ↑
[Репозитории / SheetsStore]  — только чтение/запись, без бизнес-логики
      ↑
[Сервисы]         — правила: «у детали один техпроцесс», порядок операций
      ↑
[API (Apps Script)] — HTTP envelope → модули catalog/parts/tech_process/assembly
      ↑
[Фронтенд-модули] — экраны по разделам: детали, техпроцесс, инструмент
```

**Правило:** модуль A не импортирует внутренности модуля B. Общение — через **контракты** (интерфейсы, типы данных, API).

### 2.2. Модули бэкенда (отдельные папки)

Прод: `sheets-backend/` (Apps Script). Архив FastAPI: `backend/`.

```
sheets-backend/
├── Core.gs               # конфиг, auth, envelope, ошибки
├── SheetsStore.gs        # листы, meta id
├── Catalog.gs
├── Parts.gs              # + Drive фото
├── TechProcess.gs
├── Assembly.gs
├── InitSheets.gs
└── Code.gs               # doGet/doPost, роутинг
```

- Удалить `assembly` — детали и техпроцесс работают сами.
- Добавить `history` — отдельная папка, не трогает `tech_process`.

### 2.3. Модули фронтенда

```
frontend/
├── app/                  # роутинг, оболочка Mini App
├── features/
│   ├── catalog/          # список и форма инструмента
│   ├── parts/            # список деталей, карточка, фото
│   └── tech-process/     # установы, операции, редактор
├── shared/
│   ├── api/              # клиент к API (по одному файлу на модуль API)
│   └── ui/               # кнопки, поля, списки
└── telegram/             # initData, тема Telegram — изолированно
```

### 2.4. Контракты между модулями

| Модуль | Отдаёт наружу | Не знает о |
|--------|---------------|------------|
| `catalog` | id, название, тип (tool/plate/jaw) | деталях, операциях |
| `parts` | id, номер, название, urls фото | как устроены операции |
| `tech_process` | установы, операции, ссылки на id из catalog | как рисуется UI |
| `assembly` | список уникальных инструментов/пластин/кулачков по part_id | только читает через API/сервисы |

Связь **операция → инструмент** — только `tool_id`, `plate_id` (внешние ключи). Справочник меняется — техпроцесс не переписываем.

## 3. Структура данных (минимум)

**Справочник (`catalog_items`):**

- `id`, `type` (tool | plate | jaw), `name`, `note` (опционально)

**Деталь (`parts`):**

- `id`, `number`, `title`, `created_at`

**Фото детали (`part_photos`):**

- `id`, `part_id`, `file_url` (Drive), `sort_order` (+ `drive_file_id` в Sheets)

**Техпроцесс (`tech_processes`):**

- `id`, `part_id` (один к одному)

**Установ (`setups`):**

- `id`, `tech_process_id`, `order` (I, II…), `jaw_id` (кулачки из catalog)

**Операция (`operations`):**

- `id`, `setup_id`, `order`, `op_number` (010, 020), `title`, `tool_id`, `plate_id`, `comment`

## 4. Этапы реализации (каждый — отдельно)

### Этап 0. Каркас проекта

**Цель:** пустой, но запускаемый проект. Бизнес-логики нет.

**Делаем:**

- репозиторий: `sheets-backend/`, `frontend/` (архив FastAPI — `backend/`);
- Apps Script `core`: Script Properties, health-check `GET /health` → `{status:"OK"}`;
- фронт: заглушка «Техпроцессы ЧПУ», подключение Telegram WebApp SDK.

**Проверка:** Web App отвечает на health, фронт показывает заглушку.

**Файлы:** `sheets-backend/Core.gs`, `Code.gs`, `frontend/app/App.tsx`, `telegram/init.ts`.

**Не делаем:** детали, инструмент, техпроцесс.

---

### Этап 1. Справочник инструмента (`catalog`)

**Цель:** вести инструмент, пластины, кулачки. Ни от кого не зависит.

**Делаем:**

- таблица `catalog_items`;
- API: список, создание, изменение, удаление, поиск по имени, фильтр по `type`;
- экран «Инструмент»: три вкладки или фильтр — Инструмент / Пластины / Кулачки;
- форма: название, примечание.

**Проверка:** добавил «CNMG 120408» и «Кулачки высота 15» — они в списке, поиск находит.

**Модули:** только `sheets-backend/Catalog.gs`, `frontend/features/catalog/*`.

**Изоляция:** другие этапы не трогаем.

---

### Этап 2. Детали (`parts`)

**Цель:** список деталей, карточка, фото. Без техпроцесса.

**Делаем:**

- таблицы `parts`, `part_photos`;
- загрузка фото → хранилище файлов;
- API: CRUD деталей, upload/delete/reorder фото;
- экраны: список деталей, поиск по номеру/названию, карточка с галереей;
- кнопка «+ Новая деталь».

**Проверка:** создал «В-204 Втулка», загрузил 2 фото — всё отображается.

**Модули:** `sheets-backend/Parts.gs`, `frontend/features/parts/*`.

**Зависимости:** только Этап 0. От `catalog` и `tech_process` не зависит.

---

### Этап 3. Техпроцесс — каркас (`tech_process`)

**Цель:** один техпроцесс на деталь, установы с кулачками, без операций.

**Делаем:**

- таблицы `tech_processes`, `setups`;
- API: создать техпроцесс для `part_id`, добавить/изменить/удалить установ, выбрать `jaw_id` из catalog;
- на карточке детали — блок «Техпроцесс»: Установ I, Установ II, у каждого — кулачки (выбор из справочника).

**Проверка:** у В-204 два установа, у каждого указаны кулачки.

**Модули:** `sheets-backend/TechProcess.gs`, `frontend/features/tech-process/` (только установы).

**Зависимости:** Этап 1 (кулачки), Этап 2 (деталь). Операции — следующий этап.

---

### Этап 4. Операции в техпроцессе

**Цель:** шаги внутри установа.

**Делаем:**

- таблица `operations`;
- API: CRUD операций, смена порядка (вверх/вниз);
- поля: номер оп., что делаем, инструмент, пластина, комментарий;
- выбор инструмента и пластины — из `catalog` (типы tool и plate);
- UI: таблица операций внутри каждого установа.

**Проверка:** в Установе I четыре операции (010–040), у каждой свой инструмент и пластина.

**Модули:** расширяем только `tech_process` и `features/tech-process`. `catalog` и `parts` не меняем по структуре.

**Изоляция:** операция хранит только `tool_id` / `plate_id` — удаление позиции из справочника обрабатываем отдельно (запрет удаления или «битая ссылка» — решить один раз в `catalog.service`).

---

### Этап 5. Сводка «Что нужно для детали» (`assembly`)

**Цель:** по детали автоматически список всего инструмента, пластин и кулачков.

**Делаем:**

- сервис `assembly`: по `part_id` обходит техпроцесс, собирает уникальные id из catalog;
- API: `GET /parts/{id}/required-items`;
- на карточке детали — блок «Нужно для изготовления» (только чтение).

**Проверка:** для В-204 список совпадает с тем, что в операциях и кулачках установа.

**Модули:** только `sheets-backend/Assembly.gs` + один блок UI.

**Изоляция:** модуль только читает. Выключили — карточка детали и техпроцесс работают.

---

### Этап 6. Главный экран и навигация

**Цель:** как в согласованной концепции — поиск, список, две кнопки.

**Делаем:**

- главная: поиск, список деталей (последние / по имени);
- кнопки: «+ Новая деталь», «Инструмент»;
- нижняя или боковая навигация: Детали | Инструмент;
- открытие детали → карточка (фото + техпроцесс + сводка).

**Проверка:** за 3 клика: нашёл деталь → вижу техпроцесс и список инструмента.

**Модули:** `frontend/app/` — только сборка уже готовых `features/*`.

---

### Этап 7. Telegram: авторизация и ограничение доступа

**Цель:** в API заходят только из твоего Mini App.

**Делаем:**

- фронт передаёт `initData` от Telegram WebApp (envelope / заголовок в payload);
- Apps Script проверяет подпись (`BOT_TOKEN` в Script Properties);
- опционально: белый список `TELEGRAM_ALLOWED_USER_IDS`;
- флаг `TELEGRAM_AUTH_ENABLED`; роутеры модулей не трогаем.

**Проверка:** запрос без initData → 401; из Telegram → OK.

**Изоляция:** Telegram-логика в `sheets-backend/Core.gs` и `frontend/telegram/`. Отключил — API открыт (только для разработки).

---

### Этап 8. Копирование техпроцесса

**Цель:** похожая деталь — не набивать с нуля.

**Делаем:**

- API: `POST /parts/{id}/copy-tech-process-from/{source_part_id}`;
- кнопка на карточке детали: «Скопировать с другой детали».

**Модули:** метод в `tech_process/service.py`, кнопка в UI.

**Зависимости:** этапы 2–4. Остальное не ломает.

---

### Этап 9. (Опционально) История изменений

**Цель:** когда и что меняли в техпроцессе.

**Делаем:**

- таблица `audit_log`: кто, когда, что изменил;
- запись из `tech_process/service` при сохранении;
- экран «История» на карточке детали.

**Изоляция:** отдельный модуль `history/`. Нет модуля — всё работает без истории.

---

### Этап 10. Развёртывание на GitHub

**Цель:** Mini App доступен из Telegram, деплой идёт из репозитория на GitHub.

**Делаем:**

- **GitHub Pages** — публикация фронтенда (HTTPS для Telegram уже есть);
- **GitHub Actions** — сборка `frontend/`, деплой в Pages при push в `main`;
- в `frontend` — `base` в Vite под путь репозитория (`/<repo>/` или кастомный домен);
- **GitHub Secrets** — `VITE_API_URL` = URL Apps Script Web App;
- API: деплой вручную из `sheets-backend/` как Google Apps Script Web App (Script Properties: таблица, Drive, `BOT_TOKEN`);
- в BotFather — URL Mini App: `https://<user>.github.io/<repo>/`;
- в `Readme.md` — как задеплоить Sheets backend и Pages.

**Проверка:** push в `main` → Actions зелёный → Mini App → данные из Google Таблицы.

**Файлы:** `.github/workflows/deploy-frontend.yml`, `sheets-backend/*`, `frontend/vite.config.ts`.

**Не делаем:** Render/VPS/Postgres для MVP, параллельный FastAPI «на всякий случай».

## 5. Порядок этапов (что за чем)

```
0 Каркас
   ↓
1 Справочник (catalog)     ← можно параллельно с 2 после 0
   ↓
2 Детали (parts)
   ↓
3 Техпроцесс: установы
   ↓
4 Операции
   ↓
5 Сводка (assembly)
   ↓
6 Главный экран
   ↓
7 Telegram auth
   ↓
8 Копирование ТП
   ↓
9 История (опционально)
   ↓
10 Развёртывание на GitHub
```

**Минимально рабочий продукт для цеха:** этапы **0 → 1 → 2 → 3 → 4 → 6 → 7 → 10**.

## 6. API по модулям (чтобы не слипалось)

| Модуль | Эндпоинты (пример) |
|--------|-------------------|
| `catalog` | `GET/POST /catalog`, `GET/PATCH/DELETE /catalog/{id}`, `?type=plate` |
| `parts` | `GET/POST /parts`, `GET/PATCH/DELETE /parts/{id}`, `POST /parts/{id}/photos` |
| `tech_process` | `GET/PUT /parts/{id}/tech-process`, `POST .../setups`, `POST .../operations`, `PATCH .../operations/reorder` |
| `assembly` | `GET /parts/{id}/required-items` |

Префиксы и версия (`/api/v1/`) — в `main.py` одной строкой на роутер.

---

## 7. Как не сломать при добавлении функции

| Действие | Где менять | Что не трогать |
|----------|------------|----------------|
| Новый тип в справочнике | `catalog`: enum `type` + вкладка UI | `tech_process`, `parts` |
| Поле «примечание к детали» | `parts.models` + форма | `catalog`, `assembly` |
| Экспорт техпроцесса в PDF | новый `export/router.py` | сервисы только читают через API |
| Второй станок / фрезер | новый модуль `machine_types` позже | текущие таблицы без поломок |

**Запрет:** один файл `app.py` на 2000 строк и прямые SQL-запросы из роутов — при любом изменении ломается всё.

---

## Резюме

1. **Этапы 0–10** — каждый закончен сам по себе, с своей проверкой.
2. **Модули:** `catalog`, `parts`, `tech_process`, `assembly` (+ опционально `history`) — отдельные папки на бэке и во фронте.
3. **Изоляция:** связи через id и HTTP-контракты; сводка и история только читают или пишут в свой угол.
4. **Развёртывание:** GitHub Pages (фронт) + Apps Script Web App (API) + Sheets/Drive.
5. **MVP для цеха:** каркас → справочник → детали с фото → техпроцесс с установами и операциями → главный экран → проверка Telegram → деплой на GitHub.
